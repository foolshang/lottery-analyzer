## Phase A1 — สีตามความถี่ซ้ำตำแหน่งเดิม

**เป้าหมาย:** ในรายการข้อมูลหลัก (ทุกงวดที่ใส่) ระบายสีเลขแต่ละตัวตามจำนวนครั้งที่เลขนั้นออกซ้ำในตำแหน่งเดิม

**สเปก:**
- นับ "ความถี่ของคู่ (เลข, ตำแหน่ง)" — เช่น ตำแหน่งที่ 1 มีเลข 5 ออกกี่ครั้งในทั้งชุดข้อมูล
- ระบายสีเลขในรายการตามความถี่นั้น
- **ช่วงสีปรับอัตโนมัติจากต่ำสุด→สูงสุดของข้อมูลจริง** (ห้าม hard-code 1/2-3/4-6) เช่น แบ่งเป็น 3-4 ระดับจาก min-max ที่พบ (quantile หรือแบ่งช่วงเท่ากัน)
- ระดับสี: ความถี่ต่ำ = สีกลาง/เทา → ความถี่สูง = สีเข้ม/ร้อน (เลือก palette ที่อ่านง่าย ตัดกับพื้นหลังเข้มของแอป)
- ใส่ legend เล็กๆ บอกว่าสีไหน = ความถี่เท่าไร

**ขั้นตอน:**
1. หาฟังก์ชัน/จุดที่ render รายการข้อมูลหลัก (ดู `rows.map` ใน v25 ~บรรทัด 613, และจุดแสดงผลในแท็บข้อมูล)
2. เขียน helper คำนวณความถี่ (เลข,ตำแหน่ง) จาก rows ทั้งหมด
3. เขียน helper map ความถี่ → ระดับสี (auto-scale จาก min-max)
4. ใส่สีในการ render แต่ละหลัก
5. เพิ่ม legend

**ทดสอบ:** ใส่ข้อมูลหลายงวด ดูว่าสีไล่ระดับสมเหตุผล (เลขที่ออกบ่อยในตำแหน่งเดิม = สีเข้มกว่า) และปรับ scale ตามจำนวนงวด

## Phase A2 — lockedHints เป็นข้อมูลหลัก (แยกกลุ่ม)

**เป้าหมาย:** นำ lockedHints (3หน้า/3หลัง/2หลัง) มานับเป็นข้อมูลหลักด้วย แสดงแยกกลุ่มจากเลข 6 ตัว ใช้ระบบสีเดียวกัน และยังคง boost

**สเปก:**
- แสดงรายการ lockedHints เป็น **กลุ่มแยกต่างหาก** จากรายการเลข 6 ตัว (คนละ section)
- ใช้ระบบสี A1 เดียวกัน แต่ **นับความถี่แยกกลุ่ม** (lockedHints มี scale ของตัวเอง ไม่รวมกับเลข 6 ตัว — เพราะจำนวนตัวอย่าง/ตำแหน่งต่างกัน)
  - map ตำแหน่ง: front → pos 1,2,3 ; back 3ตัว → pos 4,5,6 ; back 2ตัว → pos 5,6
- **ยังคง boost** เหมือนเดิม แต่แก้ปัญหา saturate (ดูด้านล่าง)

**⚠️ จุดที่ต้องแก้ — boost saturate (ตัดสินใจแล้ว: normalize):**
- ปัญหา: lockedHints มี ~246 ชุด ถ้า boost +3/ชุด น้ำหนักทุกตำแหน่งชนเพดาน ±20 หมด → ทำนายเพี้ยน
- **วิธีแก้: normalize** — แปลงความถี่ของแต่ละเลข (ในแต่ละตำแหน่ง) เป็นสัดส่วน 0-1 แล้วคูณด้วยเพดาน boost ที่กำหนด (เช่น boost สูงสุด +X) → เลขที่ปรากฏบ่อยได้น้ำหนักมากตามสัดส่วน แต่รวมแล้วไม่ทะลุเพดาน เลขเด่นไม่กลบเลขอื่นจนหมด
- **ทดลองค่าจริง** กับข้อมูล 246 ชุด แล้วปรับเพดาน boost ให้การทำนายยังสมเหตุผล (ไม่มีตำแหน่งไหนที่เลขเดียว dominate 100%) — รายงานค่าที่เลือกตอนสรุป

**ทดสอบ:** โหลด lockedHints จริง (246 ชุด) → ดูว่ากลุ่มแยกแสดงถูก, สีไล่ระดับ, และการทำนายไม่เพี้ยน (เทียบ weights ก่อน/หลังว่าไม่ชนเพดานทุกตำแหน่ง)

# 🅱️ ก้อน B — ระบบอัตโนมัติส่ง Discord

## Phase B1 — แยก engine ออกจาก App.jsx (หัวใจ ทำก่อน B อื่น)

**เป้าหมาย:** ดึง logic การคำนวณออกจาก App.jsx เป็น module กลาง (`engine.js`) ที่รันได้ทั้งบน browser (เว็บ) และ Node (automation) — ผลต้องเหมือนเดิมเป๊ะ

**ฟังก์ชันที่ต้องย้ายเข้า engine.js (เป็น pure functions ไม่ผูก React/DOM):**
- `buildFreq` — นับความถี่ตามตำแหน่ง
- logic boost จาก hints/lockedHints (รวม normalize ใหม่จาก A2)
- learning — ปรับน้ำหนัก ±2 (LEARN_RATE=2, MAX_W=±20)
- การทำนาย — เลือกเลขจาก freq + weight
- การตรวจผล + คำนวณ hits (6 ตัว + รางวัลย่อย)
- helper parse ข้อมูล (split บรรทัด, leading zero, ฯลฯ)

**ข้อกำหนด:**
- `engine.js` ต้องเป็น JavaScript ที่ไม่ import อะไรจาก React/browser API → รันด้วย Node ได้ตรงๆ
- ใช้รูปแบบ export ที่ใช้ได้ทั้ง 2 ฝั่ง (ESM `export` — ให้เข้ากับ build ของเว็บ และ Node `--experimental` หรือตั้ง type:module)
- App.jsx เปลี่ยนมา `import` จาก engine.js แทน logic เดิม
- **เว็บต้องทำงานเหมือนเดิมทุกอย่าง** — ทดสอบเทียบ: ใส่ข้อมูลชุดเดียวกัน ผลทำนาย/weights/hits ต้องตรงกับ v25 ก่อน refactor

**ขั้นตอน:**
1. อ่าน v25 ทั้งไฟล์ ระบุฟังก์ชันคำนวณทั้งหมด + dependency ระหว่างกัน
2. สร้าง engine.js ย้ายฟังก์ชันมา (เป็น pure function รับ input ชัด คืน output ชัด)
3. แก้ App.jsx ให้ import จาก engine.js
4. ทดสอบเทียบผลก่อน/หลัง (เขียน test เล็กๆ ด้วย Node เทียบ output)
5. เป็น version ใหม่ (lottery-app-vN.jsx) + CHANGELOG

**นี่คือ version ใหม่ของเว็บ** — bump version + ใส่ CHANGELOG ว่า refactor engine

## Phase B2 — script ดึงข้อมูล

**B2.1 ดึง hints จาก kapook (เลขเด็ดงวดหน้า)**
- 5 สำนัก: แม่น้ำหนึ่ง, แม่จำเนียร, บ้านสีฟ้า, คำชะโนด, หลวงพ่อปากแดง
- URL หน้าสำนัก: `https://lottery.kapook.com/<ชื่อสำนัก URL-encoded>` → เป็นหน้ารวมลิงก์
- flow: fetch หน้าสำนัก → หาลิงก์บทความงวดล่าสุด (`viewXXXXXX.html`) → fetch บทความ → regex จับเลข (ในตัวอย่างเป็น pattern `**- NN**` / `- NN`)
- ดึงด้วย Node `fetch()` + regex (ไม่ใช้ AI/token); ใส่ User-Agent ปกติ; เว้นจังหวะระหว่างคำขอกัน rate limit
- **การเลือกเลข (ตัดสินใจแล้ว): เก็บทุกเลข ถ่วงน้ำหนักตามจำนวนสำนักที่ซ้ำ** — รวมเลขจาก 5 สำนัก นับว่าเลขไหนปรากฏกี่สำนัก → เลขซ้ำหลายสำนักได้น้ำหนักมากกว่า (ไม่ตัดเลขที่สำนักเดียวให้ทิ้ง)
- แปลงเป็น hints format `{front, back}` แล้วเขียน Firestore (merge เฉพาะ field ที่เกี่ยว เหมือน write-hints.js เดิม)
- ⚠️ เลขเด็ดส่วนใหญ่เป็น 2 ตัว (บางสำนักมี 3 ตัว) — ต้อง map เข้า back (2หลัก) / front-back (3หลัก) ให้เหมาะ ตัดสินใจตอน implement แล้วรายงาน
- ⚠️ หวยซองไม่เอา (รวมหลายเจ้า); บ้านสีฟ้าเอา (ซองเจ้าเดียว)
- **สำรอง:** myhora `lucky-number.aspx` — เลขโหวตรายหลัก 0-9 ฝังใน URL รูป `vote-chart.aspx?count=n0,n1,...,n9` (เผื่อใช้ทีหลัง ไม่ต้องทำตอนนี้ถ้า kapook พอ)

**B2.2 ดึงผลตรวจจาก myhora (รางวัลที่ออกแล้ว)**
- URL: `https://myhora.com/lottery/result-25YY.aspx` (หน้ารายปี มีทุกงวด งวดล่าสุดบนสุด)
- แต่ละงวดมี: รางวัลที่ 1 (6 ตัว), เลขหน้า 3 ตัว (2 ชุด), เลขท้าย 3 ตัว (2 ชุด), เลขท้าย 2 ตัว (1 ชุด)
- ดึงงวดล่าสุด → 6 ตัว เข้า `data`; รางวัลย่อย เข้า lockedHints (ตามที่ใช้อยู่)
- parse ด้วย regex จาก HTML

## C1 — ตรวจว่า hints รีเซ็ตทุกงวด (เขียนทับ ไม่สะสม)

### พื้นหลัง (ทำไมต้องทำ)
ใน Firestore document `lottery/shared_data` มี 2 field ที่เก็บข้อมูลเสริม คนละบทบาทกัน:
- **`hints`** = เลขเด็ดงวดหน้า ดึงจาก kapook โดย `scripts/fetch-hints.js` → ต้องเป็นของ **งวดเดียว** พองวดใหม่มาต้อง **ล้างของเก่าทิ้งหมด** แล้วใส่ของใหม่ (ไม่งั้นเลขเด็ดงวดก่อนค้างปนกับงวดใหม่)
- **`lockedHints`** = ข้อมูลหลักสะสม 246 ชุด (ผลรางวัลย่อยย้อนหลังปี 66-69) → **ห้ามรีเซ็ต ห้ามแตะ** ต้องอยู่ครบตลอด

### ขั้นตอนที่ต้องทำ (ทำทีละข้อ)

**ขั้น 1:** เปิดไฟล์ `scripts/fetch-hints.js`

**ขั้น 2:** หาบรรทัดที่เขียนข้อมูลลง Firestore (มองหา `.set(` หรือ `.update(` หรือ `.doc(` ที่เกี่ยวกับการเขียน) — ดูว่ามันเขียน field `hints` อย่างไร

**ขั้น 3:** ตรวจว่าเข้าเงื่อนไขไหน:
- ✅ **ถูกต้องแล้ว** ถ้า: โค้ดสร้าง array ใหม่ของ hints งวดนี้ แล้วเขียนทับทั้งก้อน เช่น `set({ hints: newHintsArray }, { merge: true })` — แบบนี้เลขเด็ดงวดเก่าถูกแทนที่หมด ไม่ต้องแก้
- ❌ **ผิด ต้องแก้** ถ้า: โค้ดอ่าน hints เดิมมาก่อนแล้วต่อท้าย (เช่น `[...oldHints, ...newHints]`) หรือใช้ `arrayUnion` หรือ append แบบใดๆ — แบบนี้เลขเก่าสะสมทับ ต้องแก้

**ขั้น 4 (ถ้าผิด):** แก้ให้เขียน hints ใหม่ทั้ง array ทุกครั้ง โดยใช้รูปแบบนี้:
```js
// เขียนทับ field hints ทั้งก้อน แต่ merge:true กัน field อื่น (data/lockedHints/weights/history) ไม่ให้หาย
await db.collection('lottery').doc('shared_data').set(
  { hints: newHintsArray, updatedAt: new Date().toISOString() },
  { merge: true }
);
```
สำคัญ: ต้องใช้ `{ merge: true }` เพื่อไม่ให้ field อื่นหาย แต่ตัว `hints` จะถูกแทนค่าใหม่ทั้งก้อน (ไม่ใช่ arrayUnion ที่จะสะสม)

**ขั้น 5 (ยืนยันสำคัญ):** อ่าน `scripts/fetch-hints.js` ทั้งไฟล์ ตรวจว่ามัน**ไม่มีการเขียน field `lockedHints` เลย** — fetch-hints.js ต้องแตะแค่ `hints` เท่านั้น ถ้าพบว่ามันเขียน/แก้ `lockedHints` ด้วย = ผิด ให้เอาส่วนที่แตะ lockedHints ออก (เพราะ 246 ชุดเป็นข้อมูลหลัก ห้ามให้ scraper ไปยุ่ง)

## C2 — ตรวจว่า boost ของ lockedHints ไม่ตัน (normalize ทำงานจริง)

### พื้นหลัง (ทำไมต้องทำ)
`lockedHints` มี 246 ชุด ถ้าระบบ boost น้ำหนัก +3 ต่อทุกชุดแบบเดิม น้ำหนัก (weights) ทุกตำแหน่งจะพุ่งชนเพดาน ±20 จนหมด → การทำนายจะเพี้ยน (เลขเด่นกลบทุกอย่าง) ตอนออกแบบ (Session 6) ตกลงกันว่าต้องทำ **normalize** = แปลงความถี่เป็นสัดส่วนแล้วคูณเพดาน ให้ boost รวมต่อตำแหน่งไม่ทะลุเพดาน ต้องตรวจว่า v26 ทำ normalize จริงและได้ผล

### ขั้นตอนที่ต้องทำ (ทำทีละข้อ)

**ขั้น 1:** เปิดไฟล์ `src/engine.js` หาฟังก์ชันที่แปลง hints/lockedHints เป็นน้ำหนัก boost (น่าจะชื่อ `convertHintsToBoost` หรือคล้ายกัน) อ่าน logic ว่ามัน normalize อย่างไร

**ขั้น 2:** สร้างไฟล์ทดสอบชั่วคราว `scripts/test-boost.js` (ไฟล์ทดสอบ จะลบทีหลังก็ได้) เนื้อหา:
- โหลด lockedHints 246 ชุดจากไฟล์ `hints-input-66-69.txt` (อ่านไฟล์ แปลงแต่ละบรรทัด `front,back` เป็น object `{front, back}` แบบเดียวกับ write-hints.js)
- เรียกฟังก์ชันแปลง boost จาก engine.js ด้วย lockedHints 246 ชุดนี้
- คำนวณ weights ออกมา แล้ว `console.log` weights ของทั้ง 6 ตำแหน่ง โดยแสดงว่าแต่ละตำแหน่ง เลข 0-9 ได้น้ำหนักเท่าไร (เรียงจากมากไปน้อย)

**ขั้น 3:** รัน `node scripts/test-boost.js` แล้วดูผล:
- ✅ **ถูกต้อง** ถ้า: แต่ละตำแหน่งน้ำหนักไล่ระดับ — เลขเด่นได้น้ำหนักสูง เลขอื่นลดหลั่นลงมา **ไม่ใช่** ทุกเลขชนเพดาน ±20 เท่ากันหมด
- ❌ **ผิด (ตัน)** ถ้า: ในตำแหน่งเดียวกัน มีหลายเลขหรือทุกเลขได้น้ำหนัก ±20 (ชนเพดาน) เท่ากันหมด → normalize ไม่ทำงานจริง

**ขั้น 4 (ถ้าตัน):** แก้สูตร normalize ใน `src/engine.js`:
- หาความถี่ของแต่ละเลขในแต่ละตำแหน่ง
- แปลงเป็นสัดส่วน: `ratio = ความถี่เลขนั้น / ความถี่สูงสุดในตำแหน่งนั้น` (ได้ค่า 0-1)
- คูณด้วยเพดาน boost ที่กำหนด (เริ่มลองที่ `BOOST_MAX = 10`): `boost = ratio * BOOST_MAX`
- ผลคือเลขเด่นสุดได้ boost = 10, เลขอื่นได้ตามสัดส่วน, ไม่มีตัวไหนทะลุ — และเมื่อรวมกับ weights เดิมต้องไม่เกิน MAX_W ±20
- ปรับ `BOOST_MAX` ถ้าจำเป็นให้เห็นการไล่ระดับชัดและไม่ตัน แล้ว**จดค่าที่เลือกไว้รายงาน**

**ขั้น 5:** รัน `node scripts/test-boost.js` ซ้ำ ยืนยันว่าไม่ตันแล้ว

**ขั้น 6:** ตรวจว่าการแก้ engine.js **ไม่ทำให้เว็บพัง** — เปิด `versions/lottery-app-v26.jsx` (หรือ version ล่าสุด) ดูว่ายัง import ฟังก์ชันจาก engine.js ได้ปกติ ถ้าแก้ signature ของฟังก์ชัน ต้องแก้ฝั่งเว็บให้ตรงด้วย

**ขั้น 7:** ถ้าแก้ engine.js → ต้อง bump version เว็บใหม่ (lottery-app-vN.jsx เลขถัดไป) พร้อม CHANGELOG ระบุว่าแก้ normalize boost

# 🅳 Phase D — วิเคราะห์ผลทำนายตกลง (โชคช่วงสั้น vs logic เพี้ยน)

> งานวิเคราะห์ (ไม่ใช่แก้โค้ดทันที) — หาคำตอบว่าทำไมผลทำนาย 6 ตัวตอนแรกถูก 1-3 ตัว/งวด แต่หลังๆ เหลือ 0-1 ตัว
> เป็นเรื่องปกติตามทฤษฎีสุ่ม (baseline 0.6 ตัว/งวด) หรือ logic เพี้ยนหลัง v26 (เพิ่ม lockedHints 246 ชุด + normalize boost ใหม่)
>
> **คำสั่งสำหรับ Claude Code: ทำทุกขั้นตามลำดับ ทำต่อเนื่องจนเสร็จแล้วค่อยสรุปทีเดียว ไม่ต้องหยุดถามระหว่างทาง ขั้นนี้เป็นการ "วิเคราะห์" ก่อน — แก้โค้ดเฉพาะเมื่อพบว่า logic เพี้ยนจริง (ดูขั้น 5)**

### พื้นหลัง (สิ่งที่ต้องรู้ก่อนวิเคราะห์)
- baseline ถ้าลอตเตอรี่สุ่มล้วน = **0.6 ตัว/งวด** (ค่าคาดหวังที่ทายเลข 6 ตัวถูกโดยบังเอิญ)
- ถูก 0-1 ตัว/งวด = ปกติตามสุ่ม; ถูก 2-3 ตัวรัวๆ = มักเป็นโชคช่วงสั้น (variance สูงตอนข้อมูลน้อย)
- "regression to the mean" = ข้อมูลน้อยค่าแกว่งแรง พอสะสมหลายงวดจะลู่เข้าค่าเฉลี่ยจริง
- v26 เปลี่ยน 2 อย่างพร้อมกัน: เพิ่ม lockedHints 246 ชุดเป็นข้อมูลหลัก + normalize boost ใหม่ → ต้องแยกว่าผลตกเพราะ variance ปกติ หรือเพราะ 2 อย่างนี้ทำ logic เพี้ยน

### ขั้นตอนที่ต้องทำ (ทำทีละข้อ)

**ขั้น 1 — ดึงข้อมูลจริง:**
- อ่าน Firestore document `lottery/shared_data` (เอา field `history`, `weights`, `lockedHints`, `data`)
- อ่าน subcollection `lottery/shared_data/logs/` ทั้งหมด (เรียงตามเวลา createdAt)
- จากแต่ละ log/history entry เอา: เลขที่ทำนาย (prediction), ผลจริง (actual), จำนวนถูก (hits) ต่องวด — เรียงตามลำดับเวลา

**ขั้น 2 — คำนวณสถิติเปรียบเทียบช่วง:**
- ค่าเฉลี่ย hits ของ **ช่วงแรก** (10-20 งวดแรกที่มีข้อมูล)
- ค่าเฉลี่ย hits ของ **ช่วงหลัง** (20 งวดล่าสุด)
- ค่าเฉลี่ย hits **สะสมรวมทั้งหมด**
- เทียบทั้ง 3 ค่ากับ baseline 0.6
- ถ้าข้อมูลพอ (30+ งวด) ทำ chi-square หรือดู distribution ว่าต่างจากสุ่มอย่างมีนัยสำคัญไหม

**ขั้น 3 — ตรวจ timing (สำคัญในการแยกสาเหตุ):**
- หาว่าจุดที่ผลทำนายเริ่มตกลง อยู่ตรงงวดไหน
- เทียบกับช่วงที่ deploy v26 / เพิ่ม lockedHints 246 ชุด (ดูจาก git log หรือ updatedAt)
- ถ้าผลตก **ตรงกับ** ช่วง v26 พอดี → น่าสงสัย logic เพี้ยน
- ถ้าผล **ค่อยๆ ลดมาตั้งแต่ก่อน** v26 → น่าจะ regression to the mean (ปกติ)

**ขั้น 4 — ตรวจ logic เพี้ยน (เช็ค weights ตัน):**
- เปิด `src/engine.js` ดูฟังก์ชันแปลง boost + analyze
- พิมพ์ค่า `weights` ปัจจุบันทั้ง 6 ตำแหน่ง (เลข 0-9 แต่ละตำแหน่งได้น้ำหนักเท่าไร เรียงมากไปน้อย)
- ดูอาการผิดปกติ:
  - ทุกเลขในตำแหน่งเดียวกันชนเพดาน ±20 เท่ากันหมด = boost ตัน (normalize ไม่ทำงาน)
  - ระบบทำนายเลขเดิมซ้ำๆ ทุกงวด ไม่เปลี่ยนตามข้อมูล = เพี้ยน
- (ขั้นนี้ทับซ้อนกับ Phase C2 — ถ้าทำ C2 แล้วใช้ผลร่วมกันได้)

**ขั้น 5 — สรุปผลและตัดสินใจ:**
- ถ้าเฉลี่ยสะสมวนๆ แถว 0.6 + weights ไม่ตัน + ผลตกไม่ตรงกับ v26 → **สรุป: ปกติตามสุ่ม (regression to the mean) ไม่ต้องแก้อะไร** ระบบทำงานถูกแล้ว ผลนี้ยืนยันว่าลอตเตอรี่สุ่มจริง
- ถ้า weights ตัน หรือทำนายเลขซ้ำๆ หรือผลตกตรงกับ v26 ชัดเจน → **logic เพี้ยน ให้แก้** (ทำตาม Phase C2 เรื่อง normalize boost) แล้ว bump version เว็บใหม่ + รายงานสิ่งที่แก้

### ✅ เมื่อทำเสร็จ สรุปทีเดียวแบบนี้
- **ค่าเฉลี่ย hits:** ช่วงแรก = ? / ช่วงหลัง = ? / สะสม = ? (เทียบ baseline 0.6)
- **chi-square / distribution:** (ถ้าทำได้) ต่างจากสุ่มอย่างมีนัยสำคัญไหม
- **timing:** จุดที่ผลตกตรงกับ v26 ไหม
- **weights:** ตันหรือไม่ (แปะตัวอย่าง 1-2 ตำแหน่ง)
- **ข้อสรุป:** ปกติตามสุ่ม (regression to the mean) หรือ logic เพี้ยน
- **ถ้าแก้:** แก้อะไร / version ใหม่เลขอะไร
